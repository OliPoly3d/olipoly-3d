# Fundraiser Manager Data Dictionary and API Contract — Phase 1 Draft

This is a review draft, not deployed schema or an authorization to implement SQL. Proposed entities and fields remain governed by `FUNDRAISER_DATA_MODEL.md`.

## Naming and representation decisions

- Internal stable IDs: UUID generated by PostgreSQL.
- Owner: `owner_id uuid` referencing `auth.users(id)`; immutable after insert.
- Existing authority adapters: Orders/Production/Recipe/Finance currently expose `user_id`; commands must validate ownership rather than trusting browser input.
- Money: `numeric(12,2)` at the database boundary and integer minor units in JavaScript calculation code. Currency is a three-letter code and all rows in one settlement share fundraiser currency.
- Time: `timestamptz`; opening interval is `starts_at <= now() and now() < ends_at`; a named timezone is retained for display.
- Archive: `archived_at`; archived rows are retained and excluded from default lists.
- Concurrency: `updated_at` precondition for edits; command/RPC state guards for transitions.

## Proposed record ownership

| Record | Stores | Must not store |
|---|---|---|
| Fundraiser | Program configuration, organizer role contact, public presentation, window/status | Canonical customer, Order, production, inventory or ledger state |
| Item | Recipe reference, display/personalization rules, configured price and payout | Recipe manufacturing definition or asset bytes |
| Order link | Stable fundraiser↔Order attribution and idempotency hash | Customer master, canonical fulfillment/payment status or Order total |
| Order line/personalization | Included quantities and immutable commercial/design snapshots | Independent Order identity or production workflow status |
| Confirmation event | Organizer assertion and optional validated Finance evidence | Finance receipt/payment posting |
| Settlement/lines | Frozen reproducibility snapshot and concrete Finance reference | General ledger entries or mutable live totals |

## Command draft

All names are placeholders pending deployed schema review.

| Command | Caller | Transactional result |
|---|---|---|
| `fundraiser_owner_create` | authenticated owner | Creates one draft with server owner/audit fields. |
| `fundraiser_owner_update` | authenticated owner | Updates allowed fields only when version/status permits. |
| `fundraiser_owner_transition` | authenticated owner | Validates lifecycle guards; records actor/time. |
| `fundraiser_owner_upsert_item` | authenticated owner | Validates owned active recipe revision and commercial terms. |
| `fundraiser_catalog_read` | anonymous/owner | Returns only allowlisted public fields when publishable; owner preview is separately authorized. |
| `fundraiser_submit_order` | anonymous or owner-assisted | Validates window/item/quantities, calls approved Orders contract, snapshots lines, and returns/replays a safe receipt atomically. **Blocked on Orders contract.** |
| `fundraiser_owner_confirm_collection` | authenticated owner | Appends an audited organizer assertion; does not post Finance. |
| `fundraiser_owner_finalize_settlement` | authenticated owner | Locks included rows, verifies Finance allocations and approved accounting policy, creates immutable snapshot. **Blocked on Finance contract.** |

## Public submission input/output allowlist

Input may include public slug, opaque idempotency key, selected item IDs/quantities, per-unit personalization values, allowed fulfillment/payment selection, and minimum Order customer/contact fields required by the approved Orders contract. Monetary snapshots, owner IDs, status, Order numbers, audit actors and Finance IDs are never accepted from the caller.

A success response may contain only an opaque receipt token, a customer-safe Order display reference if approved, accepted timestamp and customer-safe summary. It must not return internal UUIDs, other customers, personalization echoes by default, private notes, payout terms, Finance data, or asset paths.

## Production projection contract

`Completed` is derived from canonical Order/Production state and regresses on Needs Reprint. `Programmed` and `Printed` require identified canonical Production evidence. Until that evidence exists, the API returns `unavailable` plus a reason, never an editable fundraiser-owned flag. Rollups expose counts (`not_started`, `partial`, `complete`, `blocked`, `not_required`) and only export `Y` for `complete`.
